# Technical Decisions

## SearchPlatform 인터페이스를 만든 이유

플랫폼마다 검색 방식, 응답 구조, API가 모두 다르다.

공통 인터페이스를 사용하여
플랫폼이 추가되어도 기존 코드를 수정하지 않는
확장 가능한 구조를 선택하였다.

---

## Musinsa를 보류한 이유

공식 API는 제공되지 않았으며,
내부 API는 브라우저 환경 의존성이 높았다.

유지보수 비용과 안정성을 고려하여
우선순위를 낮추었다.

---

## KREAM을 조사한 이유

HTML에 SSR 데이터(__NUXT_DATA__)가 포함되어 있었다.

공식 API가 없어도
검색 기능 구현 가능성이 있다고 판단하였다.

---

## 공식 API 우선 전략

역공학 기반 구현보다

공식 API를 우선 연동하여

안정성과 유지보수성을 확보하기로 결정하였다.

---

## NAVER Shopping API를 채택한 이유

NAVER Shopping Search API는
공식적으로 지원되는 REST API이며,

WebClient를 이용한 연동이 쉽고,

현재 프로젝트의 SearchPlatform 구조와도
자연스럽게 통합할 수 있었다.

따라서 상품 검색 기능은
NAVER Shopping API를 우선 적용하기로 결정하였다.

---

## SearchPlatform 예외 처리 정책

### 결정

외부 API 호출 실패는
서비스 전체 실패로 처리하지 않는다.

### 이유

SearchPlatform은
플랫폼별 독립성을 위해 만든 구조이다.

따라서

- Naver 실패
- Kream 검색
- Musinsa 검색

은 서로 영향을 주지 않아야 한다.

외부 API 예외는
로그를 남기고 빈 리스트를 반환한다.

NullPointerException은
프로그래밍 버그이므로
catch하지 않고 수정한다.

---

## SearchResponse 도입

### 문제

기존 API는 List<ProductResponse>만 반환하였다.

이 경우

- AI 분석 결과
- 상품 검색 결과

를 동시에 전달할 수 없었다.

또한 프론트엔드가 새상품과 중고상품을 직접 분류해야 했다.

---

### 결정

SearchResponse를 도입하여

- analysis
- newProducts
- usedProducts

를 함께 반환하도록 변경하였다.

새상품과 중고상품 분리는 SearchService에서 수행한다.

---

### 이유

- AI 분석 결과와 검색 결과를 하나의 응답으로 제공할 수 있다.
- 프론트엔드가 별도의 분류 로직 없이 바로 화면을 구성할 수 있다.
- UI 구조(🛍 새상품 / ♻️ 중고)와 API 구조를 일치시킬 수 있다.
- 앞으로 새로운 플랫폼을 추가해도 응답 구조를 유지할 수 있다.

---

## platform과 mall을 분리한 이유

기존에는 mall 필드 하나만 사용하였다.

하지만 네이버 쇼핑 API의 mallName은 실제 판매처를 의미하며,
검색이 수행된 플랫폼과는 다른 개념이다.

예를 들어

- platform : Naver Shopping
- mall : 트렌비

처럼 서로 다른 정보를 표현해야 한다.

따라서 검색 플랫폼과 판매처를 명확히 구분하기 위해
platform 필드를 추가하였다.

이 구조는 추후 무신사, KREAM, 번개장터 등 여러 검색 플랫폼을 추가할 때도 그대로 사용할 수 있다.

---

## 검색 후보 생성과 검색 결과 평가를 분리한 이유

기존에는 NaverShoppingService가 다음 책임을 모두 가지고 있었다.
- 검색 후보 생성
- 네이버 쇼핑 API 호출
- 검색 결과 평가
- 검색 재시도

서비스의 책임이 많아질수록 테스트와 유지보수가 어려워진다.

따라서 검색 후보 생성과 평가를 각각 독립적인 클래스로 분리하였다.

그 결과 NaverShoppingService는 검색 흐름만 제어하도록 변경하였다.

### 이유
- 단일 책임 원칙을 적용하기 위ㅜ해
- 검색 후보 생성과 결과를 각각 독립적으로 테스트 하기 위해
- 향후 다른 쇼핑 플랫폼에서도 동일한 로직을 재사용하기 위해

---

## 검색 후보 우선순위

검색 성공률을 높이기 위해 검색 범위를 좁은 것에서 넓은 것으로 확장하도록 설계하였다.

``` text
1. 브랜드 + 상품명
2. 상품명
3. Similar Products (confidence 높은 순)
4. 브랜드
```
브랜드 검색은 범위가 가장 넓기 때문에 마지막으로.

---

## SearchNormalizer를 도입한 이유

플랫폼마다 브랜드명과 상품명이 서로 다른 형식으로 저장되어 있다.

예를 들어

```text
Nike®
nike
 NIKE
```

는 모두 동일한 브랜드지만 문자열 비교만으로는 서로 다르게 인식된다.

이를 해결하기 위해 SearchNormalizer를 도입하였다.

SearchNormalizer는

- 앞뒤 공백 제거
- 소문자 변환
- 특수문자 제거
- 공백 정규화

를 공통으로 수행한다.

모든 검색 후보 생성과 검색 결과 평가는 동일한 정규화 규칙을 사용하도록 설계하였다.

### 장점

- 문자열 비교 규칙을 한 곳에서 관리
- 중복 코드 제거
- 검색 성공률 향상
- 향후 브랜드 별칭(예: 나이키 → nike) 확장 가능

---

## 검색 이력을 분석 직후 저장한 이유

### 선택한 설계

OpenAI 이미지 분석이 성공하면 검색 이력을 즉시 DB에 저장한다.

### 선택 이유

- 모든 성공한 이미지 분석 결과를 기록할 수 있다.
- 향후 검색 기록 조회 기능을 쉽게 추가할 수 있다.
- 검색 데이터 분석 및 추천 기능의 기반이 된다.
- 검색 플랫폼 로직과 검색 이력 저장을 분리할 수 있다.

### 고려했던 대안

#### 1. 상품 검색 이후 저장

장점

- 실제 검색이 성공한 경우만 저장된다.

단점

- 검색 과정에서 실패하면 분석 결과도 함께 사라진다.

#### 2. 비동기(Event) 저장

장점

- 응답 속도가 조금 더 빨라질 수 있다.
- 확장성이 좋다.

단점

- 현재 MVP 단계에서는 구조가 불필요하게 복잡해진다.

### 최종 결정

현재는 단순하고 유지보수가 쉬운 동기(JPA) 저장 방식을 사용하고,
트래픽이 증가하면 이벤트 기반 비동기 저장으로 확장한다.

---

## SearchHistoryService를 분리한 이유

최근 검색 조회 기능은 SearchService에 추가하지 않고 별도의 SearchHistoryService로 분리하였다.

### 이유

- SearchService는 이미지 분석 및 상품 검색을 담당한다.
- SearchHistoryService는 검색 기록 조회만 담당한다.
- 서비스별 책임을 명확하게 분리하여 단일 책임 원칙(SRP)을 유지한다.
- 향후 검색 기록 관련 기능(삭제, 페이징, 검색 등)이 추가되어도 SearchService의 복잡도가 증가하지 않는다.

---

## Entity 대신 DTO를 반환한 이유

API 응답으로 SearchHistory 엔티티를 그대로 반환하지 않고 SearchHistoryResponse DTO를 반환하도록 설계하였다.

### 이유

- 데이터베이스 구조를 외부에 노출하지 않기 위해
- 프론트엔드에서 필요한 데이터만 전달하기 위해
- Entity 변경이 API 변경으로 이어지는 것을 방지하기 위해
- API와 데이터베이스를 독립적으로 관리하기 위해

---

## 최근 검색을 Top3로 조회한 이유

최근 검색 API는 최신 검색 기록 3개만 반환하도록 구현하였다.

### 이유

- 현재 Flutter 와이어프레임과 동일한 개수를 제공하기 위해
- 불필요한 데이터 조회 및 응답 크기를 줄이기 위해
- MVP 단계에서는 단순한 구조를 유지하기 위해
- 향후 더보기 기능이나 페이징이 필요할 경우 쉽게 확장할 수 있도록 고려하였다.
